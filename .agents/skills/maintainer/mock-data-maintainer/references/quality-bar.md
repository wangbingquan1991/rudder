# Mock Data Quality Bar

Mock data should make the product easier to test, screenshot, or understand.
It should not feel random.

## Realism

- Use plausible names, roles, timestamps, statuses, priorities, and amounts.
- Give records relationships: owner, parent, source, target, current status.
- Include a mix of done, active, blocked, pending, failed, and future work.
- Add comments, summaries, cost rows, or activity events when they make a page
  or workflow more understandable.

## Determinism

Testing data should be repeatable:

- use stable IDs or stable lookup keys where possible
- anchor timestamps to a fixed `now`
- describe reset steps
- avoid relying on live external services
- include enough state to assert behavior without guessing

Screenshot/demo data can use readable names over opaque IDs, but should still
use fixed timestamps and deterministic content if it is generated by a script.

## Privacy

- Do not use real people, private emails, customer secrets, tokens, or copied
  production records.
- Use clearly synthetic domains such as `example.com` when emails are needed.
- Avoid fake secrets that look usable. Use placeholders like
  `sk_test_example_redacted`.

## Density

For screenshots, empty pages are usually bad. Prefer:

- 5-8 visible rows for a table
- 2-4 statuses represented
- at least one item needing attention
- one recent activity item
- one older completed item
- enough names and descriptions to show hierarchy

For tests, density should be purposeful. Include only enough records to cover
the behavior and edge cases being asserted.

## Output Hygiene

Always include:

- what the data is for
- how to run or paste it
- how to reset it
- what edge states are included

When producing code, prefer the repo's existing API clients, seed helpers, and
schema types over ad hoc SQL unless the user specifically asks for SQL.
