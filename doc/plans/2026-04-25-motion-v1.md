# Motion V1 Control-Plane Feedback

## Summary

Add a first version of Rudder's product motion language and apply it to the
highest-value operational surfaces: live agent runs, issue board status changes,
and Organization Structure. The target is not a new animation framework. The
target is clearer state feedback using lightweight CSS primitives that fit
Rudder's control-plane character.

## Problem

Rudder already has isolated motion in chat, transcript modals, dashboard
activity rows, and toasts, but motion is not yet treated as a reusable product
contract. Important operator moments still read as static updates:

- a live run starts, progresses, or finishes
- an issue moves across board states
- work appears to move through the organization graph
- a dense surface updates without showing what changed

That makes the product feel less alive than the underlying agent system, and it
weakens operator trust because state changes do not always reveal their cause.

## Scope

In scope:

- document Motion V1 rules in `doc/DESIGN.md`
- add shared CSS motion primitives with `prefers-reduced-motion` support
- apply motion to live run cards and transcript snippets
- apply motion to kanban drag/drop and live issue cards
- apply motion to org chart entry, active agents, and delegation-style graph
  paths
- add focused tests for motion hooks and reduced-motion behavior where practical

Out of scope:

- introducing Framer Motion or another animation dependency
- broad page-transition redesign
- playful or marketing-style background animation
- semantic backend changes for delegation events
- changing issue, run, or agent data contracts

## Implementation Plan

1. Add a small `ui/src/motion.css` layer for reusable classes and keyframes.
2. Import the motion layer after the existing UI stylesheet.
3. Extend `doc/DESIGN.md` with concrete duration, easing, and placement rules.
4. Add live-run motion classes to the active agent dashboard cards.
5. Add board motion classes for droppable lanes, dragged cards, dropped cards,
   and live issue indicators.
6. Add org chart motion classes for node entry, active status, zoom settling,
   and active graph edges.
7. Add or update focused UI tests for the added motion class contracts.
8. Run typecheck, tests, build, and a rendered UI check before hand-off.

## Design Notes

- Motion must clarify state changes, not decorate surfaces.
- Use fast fades, small Y movement, restrained scale, and semantic highlights.
- Continuous animation is allowed only for live/running/attention states.
- Every non-essential motion path must respect `prefers-reduced-motion`.
- Keep CSS selectors explicit; avoid broad rules that animate unrelated cards.
- The org chart delegation effect is derived from active/running agents for V1.
  It is a visual explanation layer, not a new persisted delegation event model.

## Success Criteria

- Rudder has a documented Motion V1 standard.
- Live agent run cards visibly distinguish active work without becoming noisy.
- Issue board drag/drop states show clear target and completion feedback.
- Organization Structure shows active work flowing through reporting lines.
- Reduced-motion users get the same information without movement.
- Automated coverage protects the class/data hooks used by these surfaces.

## Validation

- `pnpm --filter @rudderhq/ui typecheck`
- focused Vitest coverage for changed UI contracts
- `pnpm test:run`
- `pnpm build`
- visual verification of dashboard, issue board, and org chart surfaces

## Open Issues

- A future phase should connect org chart delegation motion to real issue
  assignment/hand-off events instead of only active agent state.
- A future phase should unify existing one-off motion classes in chat,
  transcript modals, and dashboard activity rows under these primitives.
