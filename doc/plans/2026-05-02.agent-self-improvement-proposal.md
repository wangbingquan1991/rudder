# Agent Skill Learning Loop

## Executive Summary

Rudder should define this capability as the **Agent Skill Learning Loop**.

Users do not need to understand prompt maintenance, runtime wiring, or skill file mechanics. They need to understand one thing:

> I can teach agents from real work. Agents can learn from feedback and past tasks. Approved learnings become Skills that improve future runs.

This aligns with Rudder's product identity. Rudder is an orchestration and control platform for agent work. It helps humans and agents coordinate through goals, tasks, knowledge, workflows, approvals, feedback, and governed runtime capabilities. The north-star metric is the weekly count of real agent-work loops completed end-to-end through Rudder.

This feature is about improving real agent-work loops.

The user-facing model should collapse to four concepts:

- **Feedback**: I point out what could be better in a run.
- **Learning**: Rudder extracts a stable lesson from feedback and past work.
- **Skill Update**: The lesson is proposed as a reviewable change to a Skill.
- **Agent Skill**: The runtime capability the agent actually uses in future work.

Product definition:

> Rudder lets agents learn from real work. Users give feedback on runs, Rudder turns repeated patterns into skill updates, and approved updates improve future agent runs.

## First Principles

Users do not want to manage processes. They want agent behavior to improve.

Their real expectations are:

- "I already pointed this out. The agent should not make the same mistake again."
- "The agent I coached should become more aligned with my working style."
- "I should not have to repeat the same context, standards, boundaries, and preferences."
- "The agent should grow experience like a teammate who has worked with me before."

The product should not ask users to enter a "workflow design" mindset. It should keep them in a review, coaching, and team-management mindset.

In human teams, junior engineers usually grow from tasks, feedback, retrospectives, team conventions, and repeated exposure to successful work. A teammate improves when feedback becomes working style.

Agents should work the same way.

The first principle is:

> Agent capability growth should come from feedback and retrospectives on real work, not repeated prompt editing.

Skills are the right runtime substrate because Skill maps directly to capability growth. Rudder should turn feedback and agent self-reflection into skill updates so agent capability improves through real tasks.

## Product Positioning

Position this feature as:

- Agent self-improvement infrastructure
- Agent learning from real work
- Skill improvement from run feedback

Primary product path:

```text
Review run
-> Leave feedback
-> Rudder extracts learnings
-> User approves skill updates
-> Agent uses updated skills in future runs
-> Rudder checks whether the agent improved
```

The core value is:

> Convert scattered review behavior into long-lived reusable agent capability.

## Core Concepts

### Feedback

Feedback is a specific comment on an agent run.

Feedback must bind to evidence. It should not be a floating comment.

Evidence can be:

- a run timeline message
- a tool call
- an issue status change
- an artifact or work product
- an approval request
- a handoff
- final output
- a run summary

Feedback should be lightweight:

- "Next time, confirm issue status first."
- "Update the issue before ending the run."
- "Read `AGENTS.md` before changing code."
- "This research task should not start with an implementation plan."

Feedback is concrete, immediate, and context-bound.

### Learning

Learning is the stable lesson Rudder extracts from one or more feedback items.

Feedback preserves the user's original words. Learning is the normalized behavior recommendation.

Example feedback:

```text
Do not just exit when the task is incomplete.
```

Example learning:

```text
When a task is blocked or incomplete, leave a clear issue comment explaining the blocker, attempted work, and next action before ending the run.
```

Learning is not active by default. It is a candidate lesson that can become a skill update after review.

### Skill Update

A Skill Update is a reviewable proposed change to a Skill.

It answers:

- Which Skill changes?
- Why should it change?
- What is the proposed instruction?
- Which agent or scope is affected?
- Which runs and feedback support it?
- What is the risk?
- How can it be rolled back?

Example:

```text
Update Issue Execution Skill:
Add "Before ending a run, leave one clear closeout status: done, blocked, handoff, or needs review."
Evidence: 3 runs ended without issue updates.
Scope: Engineer Agent only.
```

### Agent Skill

An Agent Skill is the runtime object. It is not merely a knowledge-base note. It is executable know-how for a class of work.

Examples:

- Issue Execution Skill
- Coding Skill
- Research Skill
- Planning Skill
- Review Skill
- Handoff Skill
- Memory Usage Skill

At runtime, the agent loads enabled Skills and uses them as instructions, examples, guardrails, and checklists.

Rudder already has organization skills, agent-private skills, bundled skills, and enabled skill selections. This proposal extends Skills from static runtime assets into a learning surface where real work can produce reviewable, versioned updates.

### Skill Reflection

Skill Reflection is the automatic learning system.

Periodically, or after every N runs, Rudder analyzes an agent's recent runs, feedback, failures, successful patterns, user corrections, and evaluation reports. It proposes Skill Updates.

Skill Reflection asks:

- Where did this agent perform poorly?
- What did users repeatedly correct?
- Which success paths should be reused?
- Which Skill needs an update?
- Should a new Skill be created?

V1 rule:

```text
Agent or system proposes.
Human approves.
Rudder applies.
```

Skill Reflection must not silently mutate production behavior in V1.

## User Mental Model

The user model should be:

- I can teach an agent.
- The agent remembers useful experience.
- Experience becomes Skill.
- Skill changes future work.

Users do not need to understand skill files, prompt injection, runtime context, revision metadata, or evaluation pipelines.

Use UI language like:

- `Give feedback`
- `Improve future runs`
- `Agent learned this`
- `Apply to Agent Skill`
- `Suggested improvements`
- `Used in this run`
- `Missed in this run`

## System Goals

1. Lower the cost of teaching agents.
   Users should leave feedback while reviewing a run, not in a complex configuration page.

2. Convert feedback into long-term capability.
   Feedback should become learning, learning should become a Skill Update, and approved Skill Updates should affect runtime.

3. Support agent self-reflection.
   Agents should periodically propose improvements based on past runs.

4. Preserve human control.
   Automated learning may propose changes, but V1 should require review before persistent behavior changes.

5. Close the evaluation loop.
   After a Skill changes, Rudder should observe whether future runs improve.

## Non-Goals

The first version should not include:

- complex organization-wide workflow management
- automatic role-level or organization-level rollout
- marketplace-style skill sharing
- fully automatic self-modification
- complex A/B testing
- forcing users to understand skill file structure
- converting every feedback item into a permanent rule

MVP only needs to prove:

> Feedback given to an agent can become a visible improvement in future work.

## Product Flow

### Flow A: User Teaches An Agent From One Run

1. The user opens an agent run.
2. They see the agent started changing code before understanding the issue.
3. They select that timeline block and click `Give feedback`.
4. A feedback tray opens.
5. The user writes:

   ```text
   Next time, before working on an issue, confirm the issue goal, current status, assignee, and blockers.
   ```

6. The feedback stays in the tray.
7. The user continues reviewing and sees the agent ended without updating the issue.
8. They select the final output and add:

   ```text
   Before ending a run, leave a clear result: done, blocked, handoff, or needs review.
   ```

9. The user clicks `Improve future runs`.
10. Rudder extracts candidate learnings:

    ```text
    Before starting issue work, check issue goal, status, assignee, and blockers.
    ```

    ```text
    Before ending a run, leave a clear closeout status: done, blocked, handoff, or needs review.
    ```

11. The review page title is `Review what this agent should learn`.
12. The user edits, deletes, merges, approves, or rejects candidates.
13. The user clicks `Apply to Agent Skill`.
14. Rudder creates an approved Skill Update and a new Skill Revision.
15. The UI confirms:

    ```text
    Engineer Agent updated its Issue Execution Skill. Future runs will use these learnings.
    ```

### Flow B: User Learns From Multiple Runs

1. The user opens Agent Detail and reviews the last 10 runs.
2. They select 5 failed or unstable runs.
3. They click `Improve from selected runs`.
4. Rudder enters review mode.
5. The left side shows selected runs. The right side shows the feedback tray.
6. The user can add feedback from multiple runs.
7. Rudder can also suggest feedback patterns:

   ```text
   3 runs ended without closeout.
   2 runs modified code before reading project instructions.
   4 runs did not explain blockers.
   ```

8. The user accepts suggestions or adds manual feedback.
9. Rudder generates proposed Skill Updates.
10. The user sees:

    ```text
    Rudder found 3 possible improvements for Engineer Agent.
    ```

11. Each improvement shows:

    - source runs
    - original feedback
    - target Skill
    - expected behavior change
    - risk and scope

12. Approved updates create Skill Revisions.

### Flow C: Agent Performs Periodic Skill Reflection

1. Every configured interval, or every N runs, Rudder triggers Skill Reflection for an agent.
2. The reflection reads:

   - successful runs
   - failed runs
   - user feedback
   - reopened issues
   - user-edited outputs
   - missing closeout actions
   - high-cost low-output context reads
   - evaluation reports

3. Reflection creates Suggested Skill Updates:

   - "Add `AGENTS.md` reading step to Coding Skill."
   - "Add closeout requirement to Issue Execution Skill."
   - "Add blocker reporting format to Handoff Skill."
   - "Add clarification rule when issue scope is ambiguous."

4. Agent Detail shows:

   ```text
   Engineer Agent has 4 suggested improvements.
   ```

5. The user approves, edits, or rejects each suggestion.

This makes agent learning visible. The agent is not just executing work; it is accumulating operational experience.

### Flow D: Verify Improvement In Later Runs

1. The agent runs again.
2. Run Detail shows `Skills used in this run`.
3. The user sees:

   - Issue Execution Skill
   - Coding Skill
   - Handoff Skill

4. Expanding a skill shows recent learning used in the run:

   ```text
   Recent learning applied: Leave closeout status before ending a run.
   ```

5. After completion, an evaluator checks:

   - Did the agent confirm issue status?
   - Did it read required project instructions?
   - Did it leave a closeout?
   - Did it handle blockers correctly?

6. Run Detail shows:

   ```text
   Followed 4/5 active learnings.
   Missed: closeout status was not written to an issue comment.
   ```

7. The user can click `Give feedback` on a missed item, starting the next learning loop.

## Agent Journeys

The system includes several agent roles. Implementation can use the same runtime or worker behind the scenes, but the product and architecture should distinguish them.

### Worker Agent

The Worker Agent performs normal work.

Journey:

1. It is woken by heartbeat, issue assignment, mention, approval follow-up, or manual action.
2. Runtime context loads issue, project context, enabled Skills, and role configuration.
3. The agent performs work.
4. Rudder records the run timeline, messages, tool calls, issue mutations, artifacts, and activity logs.
5. The agent leaves output and closeout.
6. Evaluation checks whether active Skills were followed.

Worker Agents do not need to know how learning is generated. They only need approved Skills at runtime.

### Learning Builder Agent

The Learning Builder turns feedback into learnings.

Input:

- feedback batch
- source evidence
- target agent
- target Skill if known

Responsibilities:

- infer which behavior the user wants changed
- merge similar feedback
- filter one-off context
- choose the target Skill
- generate learning candidates
- attach evidence links
- produce confidence and risk
- produce a Skill Update draft

Principles:

- Do not turn every feedback item into a rule.
- Do not generalize local task context into global behavior.
- Do not overwrite behavior the user has not approved.

### Reflection Agent

The Reflection Agent periodically reviews past work.

Input:

- recent runs
- feedback
- evaluation reports
- issue outcomes
- loaded Skill Revisions

Responsibilities:

- identify repeated failure modes
- identify repeated success patterns
- identify frequent user corrections
- detect missed steps
- detect stale Skills
- propose updates or new Skills

Reflection is a self-review mechanism. It proposes changes; it does not directly rewrite core Skills in V1.

### Evaluator Agent

The Evaluator checks whether Skills were followed.

Input:

- run timeline
- active Skills
- loaded Skill Revisions
- issue outcome

Output:

- followed learnings
- missed learnings
- Skills that did not apply
- new failure modes

Evaluator output feeds:

- Run Detail visualization
- Agent Detail learning history
- future Skill Reflection

### Manager Agent

Later versions can introduce Manager Agent review for broader rollouts.

Example:

```text
Should this Coding Skill update apply to all Engineer Agents?
```

The Manager Agent can inspect evidence, risk, and scope before recommending rollout. V1 can skip this.

## End-To-End System Links

### Link 1: Run To Feedback

The user selects evidence in the run timeline. Rudder creates a feedback draft.

The draft records:

- `org_id`
- `agent_id`
- `run_id`
- `issue_id`
- `event_id`
- `event_seq`
- `source_kind`
- `selected_text_snapshot`
- `content_hash`
- `created_by_user_id` or `created_by_agent_id`

The user can add multiple feedback items to one session. Submitting creates a Feedback Batch.

### Link 2: Feedback To Learning

The Learning Builder reads the Feedback Batch and produces learning candidates.

Each candidate includes:

- title
- instruction
- `applies_when`
- target Skill
- evidence
- confidence
- risk
- suggested scope

The user reviews candidates.

### Link 3: Learning To Skill Update

After approval, Rudder creates a Skill Update.

The update can:

- add a practice to an existing Skill
- modify an existing instruction
- disable a stale instruction
- create a new Skill when no existing container fits

MVP should prefer updating existing Skills to avoid fragmentation.

### Link 4: Skill Update To Runtime

Approved Skill Updates produce immutable Skill Revisions.

The agent's Skill binding points to `latest_approved` or a pinned revision.

Next run, the runtime loader resolves enabled Skills and materializes the selected revisions into the agent context.

### Link 5: Runtime To Evaluation

After the run, the evaluator reads:

- run timeline
- loaded Skill Revision metadata
- issue outcome
- evidence of required or forbidden actions

It writes an evaluation report that appears in Run Detail and feeds the next reflection cycle.

## Skill Architecture

Use a two-layer Skill architecture.

### Base Skills

Base Skills are stable capability containers.

Examples:

- Issue Execution Skill
- Coding Skill
- Research Skill
- Planning Skill
- Handoff Skill
- Review Skill
- Memory Usage Skill

Base Skills may be Rudder-bundled, organization-managed, imported, or agent-private. They define the general shape of a work type.

### Learned Skill Updates

Learned Skill Updates are incremental experience generated from feedback and reflection.

They usually patch an existing Skill instead of becoming a standalone Skill.

Examples:

```text
Issue Execution Skill:
Before ending a run, always leave a closeout status.
```

```text
Coding Skill:
Before editing code, read repo-level AGENTS.md and task-specific docs.
```

```text
Handoff Skill:
When handing off, include owner, current state, blocker, and next action.
```

Benefits:

- users see a small set of stable Skills
- Rudder can continuously revise Skill contents
- the system avoids dozens of fragmented one-rule Skills
- agent capability structure remains legible

## Runtime Integration

At run start, the runtime loader should:

1. Read agent role and runtime config.
2. Read issue, goal, project, and workspace context.
3. Resolve enabled Skills.
4. Resolve `latest_approved` or pinned Skill Revisions.
5. Materialize the selected Skill files.
6. Record loaded Skill Revision metadata.
7. Start the agent.

The agent should not see raw feedback records. It should see clear Skill instructions.

Example runtime Skill content:

```text
Issue Execution Skill

When starting issue work:
- Check issue goal, current status, assignee, and blockers.
- If the issue is ambiguous, ask for clarification before editing artifacts or code.

When ending a run:
- Leave one clear closeout status: done, blocked, handoff, or needs review.
- If blocked, explain blocker, attempted work, and next action.
```

The Skill turns fuzzy feedback into executable context.

## Permission And Control

V1 default rules:

- User feedback can generate Learning.
- Learning can generate Skill Updates.
- Skill Updates require user approval before persistent behavior changes.
- Agents can propose improvements but cannot permanently modify their core Skills by default.
- All Skill Updates are rollbackable.
- All Skill Updates have evidence.
- All Skill Updates have scope.

Scope options:

- current agent
- role
- project
- organization

MVP supports only current agent.

Future trust levels:

- Level 1: agent proposes Skill Update; user approves.
- Level 2: low-risk updates auto-apply with easy rollback.
- Level 3: Manager Agent approves updates for reports.
- Level 4: high-trust agents update working Skills automatically with audit logs.

This matches Rudder's autonomy direction: more autonomous agents, but visible, controlled, and reversible.

## MVP Scope

MVP must include:

1. `Give feedback` in Run Detail.
2. Feedback tray with multiple feedback items in one run.
3. Feedback Batch submission.
4. Learning Builder generation from a Feedback Batch.
5. User review, edit, and approval of Learning candidates.
6. Approved Learning updates one managed Skill for the current agent.
7. Immutable Skill Revision creation.
8. Runtime loading of the updated Skill Revision in the next run.
9. Run Detail display of Skills used in the run.
10. Basic evaluator check for active learnings.

MVP does not include:

- cross-agent automatic application
- role-level rollout
- organization-level Skill Update
- complex reflection dashboard
- A/B testing
- marketplace distribution
- fully automatic self-modification
- complex Skill graph

MVP target:

> I taught the agent, and the next run visibly improved.

- Skill Update application rate

## Risks And Mitigations

### Risk: Overfitting

The user may give feedback from one unusual run, and the system may overgeneralize it.

Mitigations:

- default scope is current agent
- sparse evidence creates low-confidence learning
- review UI shows evidence count
- role and organization rollout require extra confirmation

### Risk: Skill Pollution

If every feedback item enters a Skill, agents become noisy and inconsistent.

Mitigations:

- Learning Builder filters one-off feedback
- similar learnings merge
- stale learnings can be disabled
- Skill content remains structured

### Risk: User Burden

If feedback review is too heavy, users will not use it.

Mitigations:

- feedback input is minimal
- system infers target Skill, scope, and evidence
- user approves, edits, or rejects

### Risk: Agent Behavior Drift

Automatic optimization can push agents away from intended behavior.

Mitigations:

- V1 does not auto-apply persistent changes
- all Skill Updates are rollbackable
- evaluator checks regressions
- high-impact updates require approval

### Risk: Invisible Impact

If users cannot see that Skills were used, learning feels like a black box.

Mitigations:

- Run Detail shows Skills used
- evaluator shows followed and missed learnings
- Agent Detail shows recent improvements

## Example

Scenario: Engineer Agent handles a Rudder coding issue.

The agent starts editing code before reading `AGENTS.md` or confirming issue status.

The user selects that timeline moment and leaves feedback:

```text
For coding tasks, read the repository-level AGENTS.md and confirm project conventions before editing code.
```

Rudder generates a learning:

```text
Before editing code, read repository-level agent instructions and task-specific docs.
```

Target Skill:

```text
Coding Skill
```

The user approves. A Skill Update creates a new Coding Skill Revision.

Next time the Engineer Agent runs on a coding task, runtime loads the updated Coding Skill. The agent reads `AGENTS.md` before implementation.

After the run, evaluator checks:

- Did it read `AGENTS.md`?
- Did it follow project verification requirements?
- Did it leave a handoff or closeout note?

Run Detail shows:

```text
Applied learning: Read project instructions before editing code.
```

That is the complete loop.
