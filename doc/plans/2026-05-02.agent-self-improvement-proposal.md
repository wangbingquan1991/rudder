下面是一版 proposal，可以作为产品/工程对齐稿的第一版。整体方向是：底层用 Rudder 的 agent skill 机制把“优化后的工作流”变成可执行、可版本化、可分发的能力；顶层让用户只感知两个交互入口：SOP generation 和 feedback / annotation。这个方向也贴合 Rudder 当前定位：Rudder 本身就是 agent work 的 orchestration/control layer，核心是让 goals、tasks、knowledge、workflows、approvals、feedback 进入同一个可治理结构；同时 repo 里也已经把 bundled skills 作为运行时能力资产来管理。 

# Proposal: Annotation-driven SOP Generation for Agent Workflow Optimization

## 1. 背景与目标

现在用户在看 Agent run 的时候，往往会产生很多零散判断：

“这一步应该先查 issue context。”
“这里不该重新读整个 repo。”
“这个 agent 下次遇到类似问题应该先问用户确认。”
“这个 handoff 不清楚，应该形成固定模板。”
“这几个 run 都犯了同一个错，可以抽象成 SOP。”

这些反馈如果只停留在 comment 或 chat 里，无法稳定进入下一次 agent 执行。反过来，如果直接让用户编辑 skill 或 prompt，又会太底层，不符合大多数用户的操作心智。

因此可以把这两个想法合并成一个闭环产品：

用户在一个或多个 Agent run 中做 annotation，先不打断当前工作；annotation 可以跨 run、跨 agent、跨 issue 暂存。用户一次性提交后，系统分析这些 evidence，生成或更新一个 Agent SOP。SOP 经用户确认后，底层 materialize 成 agent skill，再通过 agent enabled skills 分发给目标 agent。之后每次 agent run 都可以加载这个 skill，按新的工作流执行，并继续被评估、反馈、迭代。

一句话：用户在顶层做 feedback 和 SOP generation；Rudder 在底层把 SOP 变成可执行 skill，并把它接入 agent runtime。

## 2. 产品定位

这个功能不是“评论系统”，也不是“prompt 编辑器”。它应该被定义为：

“把人类对 Agent 工作过程的观察，转化为可复用、可治理、可执行的 workflow improvement loop。”

它包含三个层次。

第一层是用户交互层。用户看到的是 annotation tray、feedback batch、SOP generator、SOP review、rollout。

第二层是控制平面层。Rudder 负责保存 evidence、run link、annotation、SOP draft、版本、审批、绑定、回滚、评估结果。

第三层是运行时层。Rudder 把 approved SOP materialize 成 skill package，并通过 agent enabled skills 注入给对应 agent。Agent 不需要理解“用户刚刚 annotate 了什么”，它只需要在下一次 heartbeat/run 中看到一个明确的 workflow skill。

## 3. 核心概念

Annotation 是用户在运行证据上的标注。证据可以是一段 transcript、一条 tool call、一个 assistant message、一个 final output、一条 issue comment、一个 artifact、一次 handoff、一次 approval、一个 run summary。Annotation 不等于立即生效的修改，它默认进入 draft tray。

Annotation Session 是用户正在收集的一组 annotations。它可以跨同一个 run 的多个片段，也可以跨多个 run、多个 agent、多个 issue。Session 是临时工作区，用户可以随时继续添加、删除、合并、改写。

Feedback Batch 是用户点击 submit 后形成的正式反馈包。它是 durable object，有提交人、目标 scope、annotation items、关联 run、关联 issue、期望动作。它可以触发 SOP generation，也可以只作为某个现有 SOP 的改进建议。

SOP 是人类可读的工作流规范。它描述触发条件、适用范围、前置检查、执行步骤、质量标准、失败处理、handoff 规则、禁止行为、完成信号和评估指标。每条关键规则都应该能 trace back 到 evidence。

Workflow Skill 是 SOP 的运行时实现。它通常是一个 organization skill package，里面有 `SKILL.md` 和必要 references。SOP 负责“人类理解和治理”，skill 负责“agent 执行”。

SOP Version 是一个可回滚的版本。每个版本都有 evidence、diff、生成原因、审批状态、绑定范围、运行指标。

Runtime Binding 是 SOP/skill 对 agent 的启用关系。它可以绑定到单个 agent、一组 role、一个 org subtree，或者一个 issue/workflow 类型。MVP 可以先只支持单 agent 和手动选择 agent。

## 4. 用户旅程

### Journey A：用户在单个 Agent run 中 annotation

用户打开 Run Detail。页面还是以 human-readable timeline 为主，底层 raw transcript/tool calls 可以展开。

用户发现某一步有问题，比如 agent 在没有 checkout issue 的情况下开始读代码。用户选中这段 transcript 或 tool call，点击 “Add feedback”。

右侧出现 feedback tray。系统自动记录 evidence，包括 run id、agent id、issue id、时间点、选中的 transcript range、事件类型。用户只需要写自然语言反馈，例如：

“以后处理 issue 前必须先 checkout。不要在没有 ownership 的情况下修改代码。”

用户继续看 run，又 annotate 了另一个点：

“完成后需要 issue done 或 progress comment，不能直接退出。”

这些 annotation 都暂存在同一个 session 中，不会马上触发 agent 修改。

用户点击 “Submit feedback”。系统要求用户选择目标：

“只给这个 agent”
“给这个 role 的所有 agent”
“给这个 issue/workflow 类型”
“生成新的 SOP”
“更新已有 SOP”

MVP 中默认推荐：“Generate or update SOP for this agent’s issue execution workflow”。

提交后，Rudder 创建 Feedback Batch，并启动 SOP generation flow。

### Journey B：用户跨多个 Agent run 收集反馈

用户在 Agent Detail 或 Runs 页面选择多个 run，比如最近 10 次 engineer agent 的失败 run。

系统进入 “Review runs for SOP” 模式。左侧是 run list，右侧是 annotation tray。用户可以在不同 run 之间切换，每个 run 的 annotation 都进入同一个 session。

系统也可以自动提示 pattern：

“3 个 run 都没有 close-out signal。”
“2 个 run 反复读取相同 context。”
“4 个 run 在 handoff 时没有说明 next owner。”
“1 个 run 发生 approval 后没有回写 issue。”

用户可以接受这些 suggestion，也可以手动 annotate。

用户提交后，系统生成一个 batch summary：

“这些 annotations 主要指向 issue execution SOP：checkout、context read、work chunk、close-out、blocked/handoff handling。”

用户确认后，进入 SOP generation。

### Journey C：用户生成 SOP

SOP Generator 页面展示四块内容。

第一块是 evidence summary：系统列出参与生成的 runs、annotations、issues、agents、失败/成功样本。

第二块是 workflow diagnosis：系统解释当前 agent workflow 的重复问题，比如“没有稳定入口步骤”“完成信号不一致”“handoff 文案不可追踪”。

第三块是 SOP draft。草稿结构固定，方便审阅：

触发条件：当 agent 被分配 issue、被 comment mention、收到 approval follow-up 时适用。

前置检查：确认 identity、读取 inbox、确认 wake reason、checkout issue。

执行步骤：读 issue context，做一个 bounded work chunk，记录 artifact，提交 close-out signal。

质量标准：不得无 owner 工作；不得无结果退出；如果 blocked 必须写 blocker comment；如果需要 handoff 必须说明 owner、reason、remaining work。

禁止行为：不要重复抓取完整上下文；不要绕过 approval；不要直接改 agent config；不要在没有 source issue 的情况下新建下游任务。

完成信号：issue done、progress comment、issue block、handoff comment 四选一。

第四块是 skill materialization preview。用户看到这份 SOP 会被转成一个 skill，例如 `issue-execution-sop`，并可以预览 `SKILL.md` 的关键内容。

用户可以编辑 SOP 草稿，也可以只改 top-level instruction。确认后点击 “Approve and create skill”。

### Journey D：SOP rollout

确认后，Rudder 创建或更新 organization skill package，并把 SOP version 与 skill version 关联。

用户进入 rollout step，选择启用范围：

“只启用给当前 agent”
“启用给同 role agents”
“暂不启用，只保存 SOP”
“创建 rollout issue 让 manager/CEO agent 执行”

MVP 建议先做手动启用：用户选择一个或多个 agent，Rudder 调用 agent skill sync，把这个 skill 加到目标 agent 的 enabled skills。

下一次 agent run 时，agent runtime 只会加载该 agent 明确 enabled 的 skill。这样不会把 SOP 误注入所有 agent，也符合“skill 默认可用但不默认启用”的治理心智。

### Journey E：用户观察 SOP 是否有效

后续 run detail 中会出现一个 “SOP adherence” 区域。

它回答三个问题：

这次 run 加载了哪些 SOP skills？
agent 是否遵守了关键步骤？
哪些地方偏离了 SOP，需要用户反馈？

用户可以直接在 adherence checklist 上 annotate，例如：

“这里虽然有 progress comment，但没有说明 next step。”
“这个 SOP 太严格，research task 不应该强制 checkout issue。”

这些新 annotation 又可以进入下一轮 Feedback Batch，更新 SOP v1.1。

## 5. Agent 旅程

这个 proposal 里不只是用户旅程，也要明确 agent 的旅程。这里建议把 agent 分为四类角色：Worker Agent、SOP Generator Agent、Evaluator Agent、Manager/CEO Agent。MVP 中这些角色可以由同一个 runtime agent 承担，但产品概念上要分清。

### Worker Agent 旅程

Worker Agent 是真正执行任务的 agent。

第一步，Worker Agent 被 Rudder wake up。wake reason 可能是 issue assignment、comment mention、approval follow-up、manual heartbeat。

第二步，runtime context 中包含 enabled skills。Worker Agent 看到 `issue-execution-sop` 这样的 skill 后，会按 `SKILL.md` 中的 workflow 执行。

第三步，Worker Agent 执行 SOP。它先确认身份和 context，再 checkout issue，再做 work chunk，最后输出 close-out signal。

第四步，Worker Agent 在 run 中留下 evidence。Rudder 会记录 transcript、tool call、comments、issue status change、artifact links、cost、duration。

第五步，如果 SOP 无法执行，Worker Agent 应该按 SOP 定义的 exception path 处理：comment 说明原因、block issue、handoff、或者请求 approval。

### SOP Generator Agent 旅程

SOP Generator Agent 的输入不是普通 issue，而是 Feedback Batch。

第一步，它读取 Feedback Batch，包括所有 annotation items、linked runs、selected evidence、用户目标 scope。

第二步，它做 evidence normalization，把用户自然语言反馈整理成 workflow facts：

“缺少 issue checkout”
“缺少 close-out”
“上下文获取重复”
“handoff 不可追踪”
“approval 后没有回写 issue”

第三步，它判断应该新建 SOP 还是更新已有 SOP。如果已有 SOP 覆盖这个 workflow，则生成 patch；如果没有，则生成新 SOP。

第四步，它生成 SOP draft。每条规则尽量绑定 evidence，避免凭空生成流程。

第五步，它生成 skill preview。这里不是简单把 SOP 复制进 prompt，而是转成 agent 可执行的 `SKILL.md` 结构：适用场景、步骤、命令、禁止项、完成信号、异常路径、examples。

第六步，它提交给用户 review。没有用户 approval，不应该自动 rollout。

### Evaluator Agent 旅程

Evaluator Agent 负责判断 SOP 是否真的改善了 run。

第一步，Evaluator 读取某个 run 的 transcript、events、issue mutations、loaded skills。

第二步，它读取对应 SOP 的 checklist。

第三步，它生成 adherence report：

哪些步骤满足？
哪些步骤缺失？
是否有 SOP 不适用但被强套？
有没有出现新 failure mode？

第四步，它把 report 作为 run-level insight 显示给用户，也可以自动创建 annotation suggestion。

第五步，如果多个 run 都出现同一偏差，它可以建议更新 SOP。

### Manager/CEO Agent 旅程

Manager/CEO Agent 不一定参与每次生成，但在治理链路中很重要。

当 SOP 会影响多个 agent、role 或组织级 workflow 时，系统可以创建 approval 或 review issue，交给 manager/CEO agent 评估。

Manager/CEO Agent 读取 SOP diff、evidence、rollout scope、风险说明。

如果它同意，可以建议 rollout；如果不同意，可以要求缩小范围、增加 exception、或只作为 advisory SOP 保存。

## 6. 端到端链路

### 链路一：Run annotation 到 Feedback Batch

Run Detail selection
→ Annotation Item draft
→ Annotation Session tray
→ 用户继续跨 run 添加 evidence
→ Submit
→ Feedback Batch created
→ Activity log records mutation
→ Optional issue/comment created for audit

这个链路的关键点是：annotation 不是散落在 run UI 里的临时 UI 状态，而是可以被提交、追踪、复用的控制平面对象。

### 链路二：Feedback Batch 到 SOP Draft

Feedback Batch
→ SOP generation run
→ Evidence normalization
→ Workflow clustering
→ Existing SOP matching
→ New SOP draft or SOP patch
→ Skill materialization preview
→ User review

这里要保证 generated SOP 可解释。用户必须能看到“为什么生成这条规则”，每条重要规则都应该能点回原始 annotation/run evidence。

### 链路三：SOP Draft 到 Skill

Approved SOP Version
→ Generate skill package
→ Write organization skill package
→ Register in organization skill library
→ Link SOP version to skill version
→ Optional approval gate
→ Ready for rollout

这里底层用 skill 机制承接。SOP 是治理对象，skill 是执行对象。一个 SOP 可以对应一个 skill，也可以一个 skill 内包含多个 references，但建议 MVP 中保持一对一，便于追踪和回滚。

### 链路四：Skill 到 Agent Runtime

Organization skill library
→ User selects target agent
→ Agent enabled skills updated
→ Adapter realizes enabled skill set
→ Next heartbeat/run loads skill
→ Agent follows SOP
→ Run records loaded skill metadata

这里要避免“生成 SOP 后自动污染所有 agent”。默认不自动启用，只在用户选择的 agent 上启用。未来可以支持 role-level rollout 和 experiment rollout。

### 链路五：Run Evaluation 到 SOP Iteration

Run with SOP loaded
→ SOP adherence check
→ Evaluator report
→ User annotation on deviations
→ New Feedback Batch
→ SOP patch proposal
→ New SOP version
→ Skill update / rollback / rollout

这就是完整闭环：观察 → 反馈 → SOP → skill → 执行 → 评估 → 再反馈。

## 7. 信息架构与页面设计

Run Detail 页面增加 annotation 能力。用户可以对 transcript block、tool card、assistant message、event、artifact、final result 做 annotation。右侧有 Feedback Tray，显示当前 session 中的所有 annotation。用户可以保存 draft、继续添加、提交。

Agent Detail 页面增加 “Workflow / SOPs” tab。这里显示该 agent 当前 enabled 的 SOP skills、历史 adherence、可用 SOP、建议生成 SOP 的 run clusters。

SOP Library 页面是 organization-level 页面。它显示所有 SOP，包括 draft、approved、archived、active。每个 SOP 有版本、owner、scope、linked skill、enabled agents、recent adherence score。

SOP Detail 页面显示 SOP 正文、版本历史、evidence links、skill preview、rollout 状态、metrics、feedback batches。

SOP Generator 页面是 wizard。入口可以来自 Run Detail 的 feedback submit，也可以来自 Agent Detail 的 “Generate SOP from runs”，也可以来自 SOP Library 的 “Create SOP”。

Approval / Review surface 可以复用 Rudder 已有 approval 和 issue 机制。影响范围较大的 SOP rollout 应该走 approval，而不是直接启用。

## 8. 数据模型建议

MVP 可以从这些对象开始：

`annotation_sessions`：保存用户正在编辑的 annotation tray。字段包括 org_id、created_by、status、target_scope_draft、created_at、updated_at。

`annotation_items`：保存每条 annotation。字段包括 session_id、source_type、source_id、run_id、agent_id、issue_id、event_id、transcript_range、selected_text_snapshot、feedback_type、body、severity、created_at。

`feedback_batches`：用户提交后的正式反馈包。字段包括 org_id、submitted_by、target_scope、status、summary、created_at。

`feedback_batch_items`：batch 与 annotation item 的关系。

`sops`：SOP 主对象。字段包括 org_id、slug、title、description、scope_type、status、current_version_id、created_by。

`sop_versions`：SOP 版本。字段包括 sop_id、version、body_markdown、generated_from_batch_id、skill_id、skill_version、approval_id、status、created_at。

`sop_evidence_links`：SOP rule 到 evidence 的链接。字段包括 sop_version_id、section_key、annotation_item_id、run_id、event_id、confidence。

`sop_rollouts`：记录启用范围。字段包括 sop_version_id、target_type、target_agent_id、target_role、status、enabled_at、disabled_at。

`sop_adherence_reports`：记录 evaluator 对某次 run 的检查结果。字段包括 sop_version_id、run_id、score、passed_checks、failed_checks、notes。

这套模型的重点是 traceability。SOP 不是凭空生成的文档，而是由 evidence 支撑的可版本化工作流对象。

## 10. Skill materialization 设计

SOP 被批准后，底层生成一个 organization skill。结构可以类似：

`SKILL.md` frontmatter：

name: issue-execution-sop
description: Follow the approved issue execution workflow for checkout, context, work chunking, close-out, blocking, and handoff.
sop_id: xxx
sop_version: 1.0.0
scope: agent / role / organization
generated_from_feedback_batch: xxx

正文建议包含：

“When to use this skill”

“Required workflow”

“Step-by-step procedure”

“Close-out rules”

“Blocked/handoff rules”

“Do not do”

“Examples”

“Evaluation checklist”

“Evidence notes”

如果 SOP 变更，生成新 skill version。Agent 可以选择自动跟随 latest，也可以 pin 到某个版本。MVP 建议默认跟随 approved latest，但保留 rollback 到上一版的能力。

这套方案的好处是：用户不需要理解 skill 文件，但系统最终还是把优化结果落到 skill 机制里。这样符合 Rudder 的运行时边界，也能和现有 organization skill library / agent enabled skills 体系衔接。

## 11. SOP generation 的输出质量要求

SOP Generator 不能只是总结用户反馈。它必须输出可执行规范。

每个 SOP 至少包含：

适用场景：什么时候该用，什么时候不该用。

前置条件：agent 开始前必须确认什么。

执行步骤：按顺序列出，不要太抽象。

完成标准：什么叫做 done。

异常路径：blocked、approval、handoff、缺少权限、上下文不足时怎么办。

禁止行为：用户明确反对或风险较大的行为。

评估标准：后续如何判断 run 是否遵守 SOP。

证据链接：关键规则来自哪些 annotation/run。

生成时也要做 conflict detection。例如一个 SOP 要求“每次都 checkout issue”，另一个 SOP 针对 research chat 说“不需要 checkout”。系统要提示 scope 冲突，而不是默默生成互相矛盾的 skill。

## 12. MVP 范围

第一版可以做得收敛一些。

MVP 必须有：

Run Detail annotation，支持选择 transcript/tool/event/output 并加入 tray。

Annotation session，支持跨同一 run 多次 annotate，并一次性 submit。

Cross-run selection，至少支持用户在 Agent Detail 选择多个 run 进入 SOP generation。

Feedback Batch，作为正式提交对象。

SOP Generator，生成 markdown SOP 和 skill preview。

用户 review/approve。

Materialize 为 organization skill。

手动选择 agent 启用 skill。

后续 run 显示 loaded SOP skill。

基础 SOP adherence checklist，可以先用规则/LLM 做简单评估。

MVP 暂不做：

全自动 rollout 到 role/org。

复杂 A/B test。

Marketplace 或跨 org SOP sharing。

完全自动修改 agent prompt。

复杂权限模型。

用户自定义 generation prompt。

## 13. 后续版本

V1.1 可以加入自动 run clustering。系统主动发现“这个 agent 最近 5 次 run 都有同类失败”，提示用户生成 SOP。

V1.2 可以加入 role-level SOP rollout。用户可以把 SOP 启用给所有 engineer agents，但需要 approval。

V1.3 可以加入 SOP experiment。比如 50% engineer agents 使用新 SOP，50% 保持旧 SOP，对比 close-out rate、平均 token、成功率、用户修正次数。

V1.4 可以加入 self-improving agent path。Agent 自己发现 SOP 不足时，可以提交 “SOP improvement suggestion”，但仍需要人类 review。

V2 可以做 SOP library 和 reusable workflow templates，例如 “issue execution SOP”“PR review SOP”“customer research SOP”“approval follow-up SOP”“agent hiring SOP”。

## 14. 成功指标

核心指标应该不是生成了多少 SOP，而是 workflow loop 是否真的改善。

建议指标：

每周完成的 feedback-to-SOP loop 数。

Annotation session submit rate。

SOP draft approval rate。

SOP rollout 后 agent repeated failure rate 是否下降。

Run close-out signal 缺失率是否下降。

Average unnecessary context/tool calls 是否下降。

SOP adherence score。

用户二次修改 SOP 的次数。

Rollback rate。

从 annotation 到 approved SOP 的时间。

这些指标可以和 Rudder 的 north-star “real agent-work loops completed end-to-end” 对齐：SOP generation 不是独立文档功能，而是提高 agent-work loop 成功率的机制。

## 15. 风险与对策

第一个风险是 annotation 过多，用户负担变高。对策是 annotation 要非常轻量，默认只需要选中片段和写一句自然语言；类型、scope、evidence link 由系统自动补全。

第二个风险是生成的 SOP 过拟合少数失败 run。对策是 SOP Generator 必须展示 evidence count、适用范围和 confidence；少量 evidence 只能生成 narrow-scope SOP，不应该默认组织级 rollout。

第三个风险是 SOP 与 agent 自主性冲突。对策是 SOP 不应该写成过度细节的脚本，而应该写成 workflow guardrail：必须做什么、禁止什么、完成标准是什么；具体执行仍由 agent 自己决定。

第四个风险是 skill 污染。对策是 organization skill library 中可以存在很多 SOP skill，但 agent 默认不启用；启用必须通过 agent enabled skills 显式绑定。

第五个风险是 SOP stale。对策是每个 SOP 都有 adherence metrics、last used、last updated、recent failures。长期不用或失败率高的 SOP 应该提示 review。

## 16. 推荐的第一版产品口径

我建议这个功能命名为 “Workflow Feedback” + “SOP Generator”，而不是直接叫 “Annotate Agent Runs”。

用户入口文案可以是：

“Leave feedback on this run”

“Collect feedback across runs”

“Generate SOP from feedback”

“Turn this SOP into an Agent Skill”

“Enable for selected agents”

这样顶层语言是用户能理解的 workflow/SOP/feedback，底层仍然用 Rudder 的 skill 机制承接执行。

最终形态是一个闭环：

用户观察 run
→ annotate 多个 evidence
→ 一次性提交 feedback
→ 生成 SOP
→ 审核并 materialize 成 skill
→ 启用给 agent
→ agent 按 SOP 执行
→ evaluator 检查 adherence
→ 用户继续反馈并迭代 SOP

这两个功能不仅可以结合，而且应该合在一起设计。Annotation 是输入层，SOP generation 是转化层，agent skill 是执行层，evaluation 是闭环层。
