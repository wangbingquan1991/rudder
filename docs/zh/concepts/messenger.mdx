Messenger 是 Rudder 的沟通和注意力界面。它把 Chat 对话、任务线程、失败运行、阻塞、评审提示、预算提醒和决策请求放在一个地方，但不替代任务作为执行系统。

![Messenger 注意力界面](../../images/messenger-approvals.png)

## Messenger 的工作

Messenger 回答一个 operator 问题：

> 现在什么需要我的注意，它属于哪个持久工作对象？

它应该把注意力带回行动：

| 注意力信号 | 持久下一步 |
| --- | --- |
| Agent 提问 | 回复，并把决定保留在任务上 |
| Run 失败 | 打开关联任务或 run，决定如何恢复 |
| 工作阻塞 | 写清楚缺少什么输入或 owner |
| 评审等待 | Approve、request changes 或 blocked |
| Chat 生成任务提案 | 接受、编辑或拒绝后再执行 |

## Messenger 和 Chat

Chat 是一种 Messenger 线程，适合对话式 intake 和澄清。

Messenger 更宽。它还包括围绕任务、评审、失败和系统提示的运营注意力流。

## Messenger 和任务

Messenger 不应该变成第二套任务系统。如果一个线程产生了真实工作，下一步应该回到任务：

- 创建新任务
- 更新现有任务状态
- 添加 close-out 评论
- 附加产物
- 指定 reviewer

这样即使对话滚走了，agent 工作仍然可检查。

## Messenger 和决策

Messenger 可以展示决策，但决策必须留下持久痕迹。评审用任务 reviewer 决定。受管操作用 approvals。产品判断或权限请求，则记录在关联任务或线程上。

## 好的使用场景

- Reviewer 要求修改，assignee 需要恢复工作。
- 失败 run 需要人工 triage 后再重试。
- 预算停止条件需要 operator 决定是否继续。
- Chat 创建的任务提案需要人工确认。
- 阻塞需要明确 owner 和下一步。
