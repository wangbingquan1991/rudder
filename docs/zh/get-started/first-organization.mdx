title: "创建第一个组织"
description: "详细走通 Rudder 中第一条有用的人类-agent 工作循环。"
---

组织是 Rudder 的第一层对象。人类 operator 在组织里给 agent 团队设定目标、结构、工作区，以及一套可执行、可评审、可回看的工作方式。

这篇指南会走完整个第一条有用流程：创建组织、雇佣 agent、把请求变成任务、分配负责人、运行 agent、检查输出，并用 Messenger 和 Calendar 理解发生了什么。

![组织工作界面](../../images/organization-work.png)

## 开始之前

先在本地启动 Rudder：

```bash
npx @rudderhq/cli@latest start
```

打开本地应用，并准备一个真实工作。不要用“测试 Rudder”这种空泛任务。第一条任务最好是一个可以产出、可以分配、可以评审的具体结果：

- “写一版 launch announcement，并产出可评审的 Markdown 文件。”
- “检查这个仓库，总结前三个 onboarding 阻碍。”
- “为优化文档截图写一个小型任务计划。”
- “review 这个产品页面，列出发布前必须修改的点。”

目标是创建一个能被分配、执行、评审和记录的工作对象。

## 1. 围绕真实目标创建组织

创建组织时，把目标写成运营目标，而不是 slogan。

好的第一目标：

- “交付一个本地优先的 agent 工作看板，并跑通一条端到端产品循环。”
- “运营 public beta launch，让 agent 工作可检查、可评审。”
- “每周用 agents 维护发布质量、文档和用户反馈。”

弱的第一目标：

- “使用 AI。”
- “让 agent 更高效。”
- “做个有意思的东西。”

组织目标很重要，因为任务、agent、项目和评审决定都应该能追溯到“为什么要做”。

![Rudder 仪表盘概览](../../images/board-overview.png)

## 2. 雇佣第一个 agent

大多数第一个组织只需要一个通用 operator 或工程 agent。先不要过度设计组织结构。第一个 agent 只需要足够清晰，能够接手一个真实任务。

重点设置四个字段：

| 字段 | 应该写什么 |
| --- | --- |
| Role | 稳定职责，例如 “Founding Engineer” 或 “Launch Operator” |
| Capabilities | 这个 agent 可以接什么类型的工作 |
| Runtime | Rudder 触发 heartbeat 时使用的执行后端 |
| Reporting line | 组织扩张后它向谁汇报 |

Capabilities 应该用来约束工作。例如：

> Implements small product changes, writes issue close-out comments, runs local validation commands, and asks for human input when access or product judgment is missing.

这句话告诉 Rudder 和未来 reviewer：这个 agent 应该做什么，以及什么事情不能默默自己决定。

## 3. 执行前先创建任务

任务是 Rudder 中的持久执行界面。Chat 可以用来澄清，但真正消耗 agent 时间的工作应该先变成任务。

![任务看板](../../images/issue-flow.png)

一个好的第一条任务应该包含：

- 清晰标题
- 期望结果
- 启动所需上下文
- 验收标准
- 准备好后有唯一 assignee
- 需要独立判断时有 reviewer

示例：

```md
Title: Draft the public beta launch announcement

Outcome:
Create a reviewable Markdown announcement for the public beta.

Context:
The announcement should explain local-first setup, issue-based agent work,
and why Rudder gives humans a control plane for agent teams.

Acceptance:
- includes a short headline
- includes setup command
- includes one concrete use case
- leaves open questions as bullets
```

## 4. 用状态表达下一步能不能做

状态不是装饰。它告诉人和 agent 下一步允许做什么。

| 状态 | 什么时候使用 |
| --- | --- |
| `backlog` | 请求是真实的，但范围、优先级或验收标准还没定 |
| `todo` | 工作已经明确，assignee 可以开始 |
| `in_progress` | agent 或人已经 checkout 并正在执行 |
| `blocked` | 下一步依赖权限、产品判断、外部输入或其他任务 |
| `in_review` | assignee 认为已经有可评审输出 |
| `done` | reviewer 或 owner 接受了结果和证据 |

第一个组织里先记住一个规则：只有当另一个人可以不再追问“这是什么意思”就能开始时，任务才应该进入 `todo`。

## 5. 只有一个清晰负责人时再 assign

当一个 agent 应该拥有下一步执行时，再给任务分配 assignee。

现在可以 assign 的情况：

- 结果足够具体
- agent 有上下文和运行时权限
- 重复执行会造成问题
- 你希望 heartbeat 接手它

先不要 assign 的情况：

- 问题还在探索阶段
- 多个 agent 需要拆分不同部分
- 这其实是一个决策，不是执行
- 验收标准缺失

Rudder 使用单 assignee 模型。如果两个 agent 需要协作，把工作拆成多个任务或子任务，而不是把同一个任务交给两个 owner。

## 6. 输出质量重要时加 reviewer

Reviewer 不是 approval。Reviewer 负责判断完成的工作是否足够好。

这些情况应该加 reviewer：

- 工作会修改公开文档、发布说明、代码或客户可见内容
- 输出需要产品或技术判断
- assignee 是 agent，而你希望另一个 agent 或人检查
- 关闭前需要明确的 approve/request changes 结论

低风险的小任务可以不加 reviewer，只要任务评论本身已经是足够证据。

工作准备好后，把任务移到 `in_review`。Reviewer 应留下结构化决定：approve、request changes、needs follow-up 或 blocked。

## 7. 运行 agent 并检查证据

Agent heartbeat 会唤醒 agent，让它检查分配给自己的工作、推进任务并汇报结果。

![Agent run 详情](../../images/agent-run-detail.png)

运行结束后，检查证据：

- transcript 或 run summary
- 任务评论
- 变更文件或产物链接
- 验证命令
- 截图或预览 URL
- 剩余风险

如果 run 结束但没有证据，这项工作就不可检查。让 agent 补一条 close-out 评论，或把任务移到 `blocked` 并写清楚下一步归谁。

## 8. Chat 用来 intake，不替代工作记录

当请求仍然是对话形态时，使用 Chat。它适合解释模糊请求、附加上下文、选择 agent，并生成任务提案。

![Chat 任务提案](../../images/chat-create-issue-proposal.png)

好的模式是：

1. 在 Chat 里输入混乱请求。
2. 让选中的 agent 追问或生成任务提案。
3. 批准或编辑这个提案。
4. 执行生成出来的任务。
5. 后续决策继续记录在任务上。

如果一段对话重要到需要分配、消耗预算、评审或未来回看，就应该把它转成任务。

## 9. 用 Messenger 恢复注意力

Messenger 用来发现什么需要人类注意：回复、任务线程、失败运行、阻塞、决策请求和系统提示。

![Messenger 注意力界面](../../images/messenger-approvals.png)

这些情况应该看 Messenger：

- agent 提问
- 失败 run 需要处理
- 阻塞需要人类回答
- approval 或 review 决定在等待
- Chat 创建的任务提案需要接受或编辑

Messenger 应该把你带回持久对象。如果下一步是真正的工作，就记录到任务上。

## 10. 用 Calendar 理解工作历史

Calendar 不只是日程页。在 Rudder 中，它用来查看 agent 运行和人类检查点如何占据时间。

![Calendar 工作历史](../../images/calendar-work-history.png)

用 Calendar 回答这些问题：

- 今天哪些 agent 在运行？
- 哪些任务消耗了最多时间？
- 执行后是否及时进入评审？
- agent 是并行推进，还是都卡在同一个阻塞点？
- 人类会议或检查点在哪里打断了工作循环？

最强的 Calendar 数据来自真实任务和 heartbeat runs，而不是手动创建装饰性的事件。

## 11. 把重复指令沉淀成技能

跑过几条任务后，你会看到重复操作：发布检查、预览环境、transcript 调试、文档 review、mock 数据截图等。稳定后，把这些流程沉淀成技能。

![技能库](../../images/skills-library.png)

一次性说明留在任务里。可复用流程放进技能里，让未来 agent 不需要每次复制长 prompt。

## 第一条有用循环

当你能追踪下面这条链路时，第一个组织就真的跑起来了：

1. 人类提出目标。
2. 请求变成任务。
3. 一个 assignee 拥有下一步。
4. agent run 留下证据。
5. reviewer 或 owner 接受、阻塞或要求修改。
6. Messenger 展示注意力需求。
7. Calendar 展示工作历史。
8. 任务保留最终记录。
