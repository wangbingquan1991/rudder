任务是 Rudder 把人类意图变成可检查 agent 工作的主要方式。这篇指南关注实际用户旅程：什么时候创建任务、什么时候 assign、什么时候加 reviewer、状态怎么设置，以及如何关闭循环。

![任务工作流界面](../../images/issue-flow.png)

## 默认旅程

大多数有用的 Rudder 工作遵循这条路径：

1. 在 Chat 中捕获请求，或直接创建任务。
2. 在 `backlog` 中塑形，直到结果和上下文清楚。
3. 当 assignee 不需要再讨论产品意图就能开始时，移到 `todo`。
4. 下一步属于一个 agent 或人时，分配唯一 owner。
5. Assignee checkout 后进入 `in_progress`。
6. 下一步属于其他人或外部条件时，移到 `blocked`。
7. 输出准备好被判断时，移到 `in_review`。
8. 证据被接受后，才关闭为 `done`。

## Backlog 还是 todo？

当问题真实但还需要塑形时，用 `backlog`。

好的 backlog 示例：

- “改善 onboarding docs”，但还没有验收标准。
- “调查为什么截图数据看起来很假。”
- “也许应该加一个 Calendar 页面”，但还没定义它回答什么用户问题。

当工作已经可以执行时，用 `todo`。

好的 todo 示例：

- “把安装截图命令替换成 `npx @rudderhq/cli@latest start`。”
- “新增 Calendar 概念页面，解释 run-history 使用场景。”
- “用非空 agent transcripts 重新生成 docs 截图。”

## 什么时候 assign

当一个 owner 应该执行下一步时，再 assign。

可以立即 assign 的情况：

- 任务在 `todo`
- owner 有需要的 runtime、仓库或上下文
- 重复执行会浪费时间或产生冲突
- 你希望 heartbeat 自动接手

先不要 assign 的情况：

- 工作还模糊
- 多个 owner 需要拆分不同部分
- 下一步其实是人类决策
- 权限或凭据缺失

Rudder 有意保持一个任务只有一个 assignee。如果 Growth、Engineering 和 QA 都要参与，就创建多个任务或子任务。

## 什么时候加 reviewer

当关闭任务需要独立判断时，加 reviewer。

适合 reviewer 的工作：

- 公开文档、发布说明、营销文案和 UX copy
- 影响用户可见流程的代码改动
- 数据迁移、发布操作和预算敏感工作
- agent 生成、需要产品判断的 proposal
- “done” 应该表示“被接受”，而不是“尝试过”

可以不加 reviewer 的工作：

- 低风险记录整理
- 会变成另一个任务的探索笔记
- close-out 评论已经足够作为证据的小型内部修复

Reviewer 是任务级路由信息，本身不是 approval gate。有输出可以判断时，再把任务移到 `in_review`。

## 状态最佳实践

| 状态 | 什么时候用 | 避免 |
| --- | --- | --- |
| `backlog` | 工作已捕获但还不可执行 | 把已经 ready 的工作长期停在这里 |
| `todo` | 任务清楚，可以分配 owner | 把模糊请求推进 agent 队列 |
| `in_progress` | 一个 owner 正在执行 | 让废弃工作一直活跃 |
| `blocked` | 下一步依赖其他人或外部条件 | 把它当成通用失败状态 |
| `in_review` | 输出准备好让 reviewer 判断 | 把半成品送审 |
| `done` | 证据被接受，没有下一步 | 没有可检查输出就关闭 |

## 如何阅读下面的用例

下面不是抽象任务板模板，而是从真实 Rudder operator 工作模式中脱敏改写出来的公开英文演示数据。阅读重点是 issue routing：来了什么信号，应该沉淀成哪条持久任务，下一步归谁，谁判断结果，以及什么证据可以关闭循环。

| 场景 | 初始信号 | 持久任务 | Assignee | Reviewer | Close-out 证据 |
| --- | --- | --- | --- | --- | --- |
| Chat 图片附件 bug | 用户在 transcript 里看到带认证信息的下载命令 | Optimize Chat image attachment handling for Codex runtime | 工程 agent | 人类/operator reviewer | 修改前后 transcript、runtime handoff 改动、回归测试 |
| Backlog 唤醒语义 | 评论 mention 了 designer，却唤醒了 engineer assignee | Backlog issue comment should not auto-wakeup assignee | 工程 agent | 控制面 reviewer | backlog 评论、显式 mention、非 backlog 任务的 wakeup 测试 |
| Chat 结构化提问 | Agent 继续前需要一到三个阻塞性选择 | Add `ask_user` question component to Chat | 工程 agent | 产品 reviewer | 活跃面板、已回答历史、下一轮 assistant turn |
| 文档站整改 | 官网文档需要 use-case-led 页面和真实截图 | Ship public Rudder docs that explain why and how to use the product | 拆成子任务 | Docs/product reviewer | preview URL、截图、验证命令、剩余风险 |
| Windows 安装验证 | 安装命令必须在 macOS 之外验证 | Test `npx` install command on Windows with Claude Code | 有 Windows 环境的 owner | Release reviewer | OS 版本、终端输出、截图、命令是否修改 |

## Use case：Chat 图片附件 bug

用户在 Chat 里发了一张截图，然后问：为什么 Codex agent 还要在 transcript 里运行一条可见的 `curl ... Authorization ...` 命令才能读取图片？这不是泛泛的“优化附件体验”，而是一个具体的信任问题：transcript 暴露了实现细节，让用户觉得 agent 原生看不了图。

1. 把 Chat 反馈沉淀成 bug 任务：“Optimize Chat image attachment handling for Codex runtime”。
2. 只有在证据还没收集完时，才放在 `backlog`：截图、受影响 runtime、当前 prompt、失败的用户旅程。
3. 期望行为清楚后移到 `todo`：图片附件应该先被 Rudder 准备给 runtime，用户可见 transcript 里不应该出现带认证信息的下载命令。
4. 分配一个工程 agent，因为下一步会跨 server prompt、runtime handoff 和测试。
5. 加 reviewer，因为这会改变用户可见 Chat 流程和 transcript 隐私预期。
6. 只有当任务里有证据时才进 `in_review`：prompt 行为、runtime bridge、transcript 截图、测试证明原始下载命令已经消失。
7. Reviewer 接受行为和证据质量后，才能关闭为 `done`。

## Use case：Backlog 评论唤醒了错误 agent

一个 backlog 任务已经挂着 engineer assignee。Operator 评论时只 mention 了 designer，但 engineer 也被唤醒开始运行。这里的问题不只是通知噪音，而是 `backlog` 的含义被破坏了。

1. 创建高优先级 bug 任务：“Backlog issue comment should not auto-wakeup assignee”。
2. 原来的 backlog 任务只作为证据，不要直接在那里面修。
3. 规则清楚后移到 `todo`：backlog 普通评论不能唤醒 assignee；显式 mention 只唤醒被 mention 的 agent。
4. 分配给负责 wakeup 语义的工程 agent。
5. 加 reviewer，因为它影响“agent 什么时候允许开始运行”的控制面不变量。
6. 只有在复现无法确认、或状态语义还没决定时，才使用 `blocked`。
7. 当测试覆盖 backlog 普通评论、显式 mention、非 backlog assignee wakeup 后，进入 `in_review`。

## Use case：Chat 需要结构化提问组件

Operator 希望 Chat 像 Codex 一样，在缺少关键决策时弹出一到三个结构化问题，让用户选择后继续。产品决策很具体：只使用一个持久化 kind，`ask_user`，不要同时出现 `ask_user` 和 `user_input_request` 两套命名。

![Chat 任务提案](../../images/chat-create-issue-proposal.png)

1. 产品契约还没定时先放 `backlog`：payload 结构、未回答状态、历史显示方式、非目标。
2. 验收标准足够具体后，移到 `todo`。
3. 分配一个工程 agent，因为 schema、server 解析和 UI 渲染需要一致落地。
4. 加人类 reviewer，因为这是用户交互行为，不只是代码管道。
5. 活跃面板、已回答历史状态、后续 assistant turn 都能看到时，移到 `in_review`。
6. Reviewer 接受 UX 后再关闭：这个组件应该只在缺少阻塞性决策时出现，而不是把所有普通澄清都组件化。

## Use case：带截图的文档站整改

需求是搭建公开官网文档：双语导航、结合 use case 的讲解、产品截图。它应该是一个 parent issue，而不是一个巨大的单 assignee 任务。

1. Parent issue 只描述最终结果：“ship public Rudder docs that explain why and how to use the product”。
2. 拆成可以独立关闭的子任务：
   - 内容结构和双语文档
   - 英文产品截图
   - 导航和下一步链接
   - 视觉 QA 和 broken-link 验证
3. 每个子任务只有一个 owner。不要把 writer、截图生产者、QA reviewer 都塞到同一个任务里当共同 assignee。
4. 需要判断质量的地方加 reviewer：公开定位、截图质量、Mintlify 导航。
5. 子任务都有 close-out 证据后，parent 才进入 `in_review`。
6. 最终 docs preview、截图、验证命令和剩余风险都链接回 parent 后，再关闭 parent。

## Use case：Windows 安装验证

任务内容是：“在 Windows 下测试 `npx` 安装命令，并用 Claude Code 跑起来。” 这件事很具体，但不应该自动分配给本地 macOS agent。

1. 测试命令和期望结果已知时，可以放进 `todo`。
2. 只有当某个人或 agent 真有 Windows 环境时，才 assign。
3. 如果没有合适 runtime，就保持未分配或分配给人类 owner。
4. 缺少 Windows 机器、凭据或安装包时，移到 `blocked`。
5. Close-out 证据要包含：终端输出、OS 版本、实际命令、失败截图（如果有），以及 docs 命令是否需要修改。

## Close-out checklist

标记 `done` 前确认：

- 输出可以从任务中看到
- 验证证据已经包含
- 有 reviewer 时，reviewer 决定已记录
- 剩余风险已说明
- 后续工作已创建，或明确决定不做
- Calendar 和 activity history 指向真实工作，而不是孤立对话
